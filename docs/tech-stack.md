## Frontend Stack

| Tech | Purpose |
|------|---------|
| **React 19 + TypeScript** (Vite) | Core framework. TypeScript everywhere — the deterministic engine and data schemas depend on type safety. |
| **Vanilla CSS** | Styling with CSS custom properties |
| **Lucide React** | Icon library |
| **Framer Motion** | Complex animations |
| **React Router** | Page routing |
| **zustand** | Game state management |
| **Vitest** | Unit + Monte Carlo balance tests for the engine |
| **Web Worker** | Runs the career simulation off the main thread |

### Game State Persistence

Game state (managed in a zustand store) is saved to `localStorage` after each roll. If the user returns, they're prompted: "Resume your build? (Roll 6/10)" or "Start fresh". Completed careers are also cached for the results screen.

## Data Layer

- Static JSON files generated by the **TypeScript data pipeline** (`scripts/data/build.ts`, run via `npm run data` / `tsx`; see [Data Strategy](data-strategy.md)) and emitted to `public/data/`.
- The app lazy-fetches these JSON files at runtime (`src/data/loadGameData.ts` / `useGameData.ts`); it never calls a live API.
- Player headshots from the NBA.com CDN (`https://cdn.nba.com/headshots/...`), with a local silhouette fallback.
- No backend — simulation runs entirely client-side in a Web Worker.
- `nba_api` (Python) is an **optional, local-only refresh** of cached raw CSVs (`requirements-dev.txt`) — stats.nba.com is IP-blocked in CI/sandboxes. The primary inputs are openly-licensed GitHub datasets + a hand-curated 2K-style ratings set (`data-source/2k/`).

## Share URL Encoding

Share URL format: `12-0.app/s/{encoded}` where `{encoded}` is a base64-compressed JSON of:
```json
{
  "attributes": ["9 player IDs + category assignments"],
  "franchise": {"id": "GSW", "decade": "2010s"},
  "difficulty": "normal",
  "seed": 12345,
  "dataVersion": "2026.06.06"
}
```
The seed + `dataVersion` allow exact replay of the career simulation. Total URL should stay under 200 characters. If the live `dataVersion` no longer matches the link, the app shows the archived result payload instead of silently recomputing (see [Data Strategy → Runtime Data Flow](data-strategy.md#runtime-data-flow)).

## Deployment

- **Vercel** — project `12-0` (team `rodriguezvillaloboshugo-2034s-projects`), live at `https://12-0-delta.vercel.app`.
- **GitHub repo:** `hugol05/12-0` (push to `main` after verifying typecheck/test/build).
- **Git integration is not connected** — the Vercel account can't see/link the `hugol05/12-0`
  repo (`vercel git connect` fails: "Make sure ... you have access to the repository"). Until the
  Vercel GitHub App is granted access to the `hugol05` account/repo (via
  github.com/settings/installations or the Vercel project's Git settings), pushes do **not**
  auto-deploy. Deploy manually after pushing: `npx vercel deploy --prod` from the repo root
  (already linked via `.vercel/project.json`).
